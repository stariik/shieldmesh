## 1. System Architecture Overview

ShieldMesh is a three-layer decentralized system: an edge intelligence layer (on-device AI + mesh networking), a client application layer (web + mobile), and a settlement layer (Solana blockchain). The architecture is designed so that the edge layer operates independently of the settlement layer, with the client layer bridging between them when connectivity permits.

## 2. Component Descriptions

### 2.1 AI Scanner Engine

| Attribute | Detail |
|---|---|
| **Technology** | TensorFlow Lite (on-device inference) |
| **Location** | Runs entirely on the user's device — no cloud inference |
| **Input** | URLs, SMS bodies, email content, wallet addresses, transaction payloads |
| **Output** | Structured threat assessment: `{ threat, category, severity, ai_score, threat_hash }` |
| **Model Size** | Target < 10 MB for mobile deployment |
| **Latency** | < 200ms inference on mid-range devices |
| **Update Mechanism** | Model weights pulled from IPFS/CDN when connectivity is available |

The AI Scanner is the first processing stage. When a user submits content for analysis, the TFLite model classifies it into one of six threat categories and assigns both a severity level (0-3) and confidence score (0-100). The scanner also generates a deterministic SHA-256 hash of the normalized payload, which serves as the unique on-chain identifier for the threat.

The scanner operates without any network dependency. It uses pre-trained weights bundled with the application, ensuring that threat detection works identically whether the device is online, offline, or operating within a mesh cluster.

### 2.2 Pollinet Mesh Network Layer

| Attribute | Detail |
|---|---|
| **Technology** | Pollinet SDK |
| **Transport** | Bluetooth Low Energy (BLE) + Wi-Fi Direct |
| **Range** | ~100m (BLE), ~200m (Wi-Fi Direct) |
| **Relay Hops** | Up to 5 hops per alert (configurable TTL) |
| **Peer Capacity** | Up to 200 concurrent peers per mesh cluster |
| **Deduplication** | Bloom filter on threat_hash prevents duplicate relay |

The mesh layer is ShieldMesh's core differentiator. When a threat is detected on one device, the alert packet (containing threat_hash, severity, ai_score, reporter signature, and hop metadata) is broadcast to all reachable peers. Each peer validates the alert signature, checks its local Bloom filter for duplicates, and re-broadcasts to its own peers up to the TTL limit.

The mesh layer also manages the offline settlement queue. Each device maintains a local list of verified-but-unsettled threats. When any device in the mesh cluster regains internet connectivity, it acts as a "gateway" and batch-settles all queued threats on Solana in a single burst of transactions.

### 2.3 Client Application (Next.js 15)

| Attribute | Detail |
|---|---|
| **Framework** | Next.js 15 with Tailwind CSS 4 |
| **State** | Zustand for client-side state management |
| **Offline** | Service worker for asset caching; IndexedDB for data persistence |
| **Theme** | Dark cybersecurity-themed UI designed for security operators |

The web client provides the primary user interface with four main views:

1. **Threat Feed** — Real-time scrollable feed of detected and reported threats, with severity indicators, AI scores, and verification status.
2. **Staking Dashboard** — Pool overview showing total staked, staker count, reward rates, and the user's personal staking position.
3. **Mesh Status** — Visualization of the local mesh cluster: connected peers, signal strength, hop counts, and alert propagation.
4. **Bounty Tracker** — List of the user's reported threats with bounty amounts, claim status, and settlement timestamps.

All accounts are Program Derived Addresses (PDAs) with deterministic seeds, ensuring that any client can derive account addresses without on-chain lookups.

### 2.5 Wallet Integration Layer

| Attribute | Detail |
|---|---|
| **Adapter** | Solana Wallet Adapter |
| **Supported Wallets** | Phantom (primary), Solflare, Backpack |
| **Operations** | stake, unstake, report_threat, verify_threat, claim_bounty |

The wallet layer handles all user authentication and transaction signing. ShieldMesh never accesses private keys directly — all signing is delegated to the user's wallet extension or mobile app. This ensures that the user maintains full custody of their funds at all times.

### 2.6 Offline Storage Layer

| Attribute | Detail |
|---|---|
| **Web** | IndexedDB via idb wrapper |
| **Mobile** | SQLite via expo-sqlite |
| **Data Stored** | Threat history, scan results, mesh alerts, transaction queue, AI model cache |

The storage layer ensures data persistence across sessions and network states. Critical data structures stored offline include:

- **Threat cache** — All threats seen (locally detected + mesh-relayed), indexed by threat_hash
- **Transaction queue** — Pending on-chain transactions awaiting connectivity
- **Mesh peer registry** — Known peers with connection history and reputation
- **AI model weights** — TFLite model binary for offline inference
- **User preferences** — Dashboard configuration, notification settings

## 5. Technology Justifications

### 5.1 Why Solana?

| Requirement | Solana Capability | Alternative Limitation |
|---|---|---|
| **Microbounty economics** | Transaction fees < $0.001. A 0.01 SOL bounty (LOW severity) remains economically viable. | Ethereum L1 fees ($1-50) make microbounties impossible. L2s add complexity and bridging latency. |
| **Sub-second finality** | ~400ms slot time with single-slot finality. Bounties settle before a device goes offline again. | Bitcoin: 10min blocks. Ethereum: 12s blocks + 2 epoch finality (~13 min). |
| **Batch settlement** | 65,000 TPS capacity. Settling 1,000 queued threats in a burst takes < 1 second of network capacity. | Most chains cannot handle burst workloads without congestion-based fee spikes. |
| **PDA architecture** | Deterministic account derivation from seeds. Clients can compute account addresses offline without RPC. | Account-model chains without PDAs require on-chain lookups to find associated data. |
| **Anchor framework** | Type-safe instruction handlers, automatic (de)serialization, built-in constraint validation. | Raw Solana program development is error-prone and verbose. |

### 5.2 Why Anchor 0.32?

| Feature | Benefit to ShieldMesh |
|---|---|
| `#[account]` derive macro | Automatic Borsh serialization/deserialization for all 4 account types |
| `#[derive(InitSpace)]` | Compile-time account size calculation prevents rent miscalculation |
| `init` / `init_if_needed` | Declarative account creation with automatic rent-exempt balance checks |
| PDA seeds + bump | Deterministic address derivation with built-in bump caching |
| `#[error_code]` | Custom error enum with human-readable messages for all 11 error cases |
| Constraint macros | Declarative access control (e.g., `constraint = staker_account.owner == staker.key()`) |

### 5.3 Why TensorFlow Lite?

| Requirement | TFLite Capability |
|---|---|
| **On-device inference** | Optimized for mobile and edge devices. No cloud dependency. |
| **Model size** | Quantized models < 10 MB. Fits on budget Android devices with limited storage. |

## 6. Offline-First Design Principles

ShieldMesh applies six offline-first design principles that govern every architectural decision.

### Principle 1: Local-First Computation

> **Every operation that can run on-device MUST run on-device.**

The AI scanner performs inference locally using TensorFlow Lite. Threat classification, hash generation, severity assignment, and confidence scoring all execute without any network call. The user receives a threat assessment in < 200ms regardless of connectivity status.

**Implementation:** TFLite model weights are bundled with the application binary. The scanner module has zero network imports and zero async fetch calls.

### Principle 2: Optimistic Local State

> **The local database is the source of truth until proven otherwise by on-chain data.**

When a user reports a threat, the threat is immediately added to the local database with status = PENDING and displayed in the threat feed. The on-chain transaction is queued for later execution. The user sees immediate feedback without waiting for Solana confirmation.

**Implementation:** Zustand store updates synchronously on user action. IndexedDB persists the state. When connectivity returns, the sync protocol reconciles local state with on-chain state, handling conflicts (e.g., duplicate threat_hash) gracefully.

### Principle 3: Mesh as Transport Layer

> **The mesh network is a transport layer, not a trust layer. Trust comes from cryptographic signatures and on-chain consensus.**

Pollinet mesh relays carry signed alert packets. Any device can relay, but receivers verify the reporter's Solana keypair signature before displaying or re-relaying the alert. This prevents man-in-the-middle injection of fake alerts into the mesh.

**Implementation:** Each alert packet includes: `{ threat_hash, severity, ai_score, reporter_pubkey, signature, hop_count, ttl }`. The signature covers the hash, severity, and ai_score fields. Receivers call `ed25519.verify()` locally before processing.

### Principle 4: Gateway Settlement

> **The first device to regain connectivity settles all pending transactions for the entire mesh cluster.**

Rather than requiring each reporter to individually submit their transactions, any device with queued settlements can act as a gateway. When it detects connectivity, it drains its local queue and submits all pending `report_threat`, `verify_threat`, and `claim_bounty` transactions to Solana.

**Implementation:** The Mesh Routing Agent monitors `navigator.onLine` (web) or `NetInfo` (React Native). On connectivity detection, it triggers the settlement pipeline: dequeue -> sign -> submit -> confirm -> clear queue -> broadcast confirmation via mesh.

### Principle 5: Graceful Degradation Matrix

> **Every feature has a defined behavior for every connectivity state.**

| Feature | Online | Offline (isolated) | Offline (mesh) |
|---|---|---|---|
| Threat scanning | Full (AI + cloud verify) | Full (AI only) | Full (AI only) |
| Threat alerting | Real-time push | Local notification | Mesh broadcast |
| Threat reporting | Immediate on-chain | Queued locally | Queued + mesh relay |
| Threat verification | Immediate on-chain | Queued locally | Queued + mesh relay |
| Bounty claiming | Immediate on-chain | Queued locally | Queued, gateway settles |
| Staking/Unstaking | Immediate on-chain | Queued locally | Queued locally |
| Threat feed | On-chain + local | Local cache only | Local + mesh-received |
| Dashboard stats | Live from RPC | Last-known cached | Last-known + mesh updates |

### Principle 6: Conflict Resolution via Deterministic PDAs

> **On-chain account addresses are deterministic. Duplicate submissions are idempotent failures, not data corruption.**

Because ThreatAccount PDAs are derived from `["threat", pool, threat_hash]`, two devices that independently detect the same threat will attempt to create the same PDA. The second transaction will fail with an "account already exists" error. This is handled gracefully: the client discards the failed transaction from its queue and updates the local threat record with the on-chain version.

**Implementation:** The sync protocol wraps each queued transaction in a try/catch. `AccountAlreadyExists` errors are treated as successful syncs (the threat was already reported by another device). All other errors trigger retry with exponential backoff.

## 7. Security Architecture

### 7.1 Threat Model

| Attack Vector | Mitigation |
|---|---|
| **Fake threat reports (bounty farming)** | Requires 2 independent validators with active stakes. Self-verification blocked on-chain. AI score provides quality signal. |
| **Sybil attack on validation** | Validators must hold staked SOL (economic cost). Each validator needs a unique keypair and stake. |
| **Mesh alert spoofing** | All mesh-relayed alerts are signed by the reporter's Solana keypair. Signature verification before display/relay. |
| **Pool drainage** | On-chain rent-exempt minimum enforced. Bounty payouts cannot reduce pool below rent exemption. |
| **Replay attacks** | Threat_hash PDAs are unique. Attempting to report the same hash twice fails at account creation. |
| **Key theft** | ShieldMesh never handles private keys. All signing delegated to Phantom/Solflare/Backpack wallet. |
| **Model poisoning** | AI model updates are pinned to verified IPFS hashes. Model integrity verified before loading. |
| **Transaction manipulation** | All transactions signed by the user's wallet. Anchor's constraint system validates all state transitions on-chain. |

### 7.2 On-Chain Validation Summary

Every instruction enforces validation rules at the Anchor constraint level and within handler logic:

```
initialize_pool:
  - authority must sign
  - pool PDA must not already exist

stake:
  - amount > 0
  - staker must sign
  - pool PDA must exist

unstake:
  - amount > 0
  - amount <= staker_account.amount
  - pool lamports - amount >= rent_exempt_minimum
  - staker_account.owner == staker.key()

report_threat:
  - severity <= 3
  - ai_score <= 100
  - threat PDA must not already exist (unique hash)
  - bounty PDA must not already exist

verify_threat:
  - validator != threat.reporter (no self-verification)
  - threat.status == 0 (PENDING only)
  - staker_account.amount > 0 (must have active stake)
  - staker_account.owner == validator.key()

claim_bounty:
  - threat.status == 1 (VERIFIED)
  - threat.validator_count >= 2
  - bounty.claimed == false
  - bounty.reporter == reporter.key()
  - threat.reporter == reporter.key()
  - pool lamports - payout >= rent_exempt_minimum
```
