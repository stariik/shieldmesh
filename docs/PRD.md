## 1. Product Overview

| Field | Detail |
|---|---|
| **Product Name** | ShieldMesh |
| **One-Liner** | Offline-first decentralized threat intelligence powered by on-device AI, mesh networking, and Solana bounties. |
| **Target Users** | Security researchers, rural and underserved communities, DeFi protocol teams, NGOs operating in censored regions, mobile-first users in connectivity-challenged environments. |
| **Core Problem** | 3.4 billion people live in regions with unreliable or compromised internet connectivity. Every existing cybersecurity platform — from antivirus to SIEM — requires a persistent internet connection. When authoritarian regimes cut access, when disasters destroy infrastructure, or when adversaries deliberately target connectivity, all cloud-based security tools go dark at the exact moment they are needed most. Phishing, social engineering, and crypto scams do not stop when the network goes down; they accelerate. There is no offline security layer. |
| **Solution** | ShieldMesh turns every smartphone and laptop into a threat intelligence node that works with or without the internet. On-device AI (TensorFlow Lite) detects threats locally. Pollinet mesh networking relays alerts device-to-device via Bluetooth and Wi-Fi Direct. Community members stake SOL into bounty pools, and verified threat reporters earn on-chain rewards. The first device that regains connectivity settles all pending threats on Solana in a single batch. |
| **Chain** | Solana (Devnet, targeting Mainnet) |
| **Framework** | Anchor 0.32 |

| FR-AI-05 | Severity mapping | LOW (0): suspicious pattern. MEDIUM (1): confirmed phishing. HIGH (2): active credential harvesting. CRITICAL (3): zero-day or coordinated campaign. |

### 4.2 Pollinet Mesh Networking Module

| ID | Requirement | Detail |
|---|---|---|
| FR-MESH-01 | Transport protocols | Support Bluetooth Low Energy (BLE) and Wi-Fi Direct for device-to-device communication. |
| FR-MESH-02 | Alert broadcast | When a threat is detected locally, broadcast a signed alert packet to all reachable mesh peers. |
| FR-MESH-03 | Multi-hop relay | Alerts must propagate through intermediate devices (minimum 3-hop relay) to extend coverage beyond direct radio range. |
| FR-MESH-04 | Deduplication | Each alert carries the threat_hash. Devices must discard duplicate alerts they have already received or relayed. |

| FR-MESH-06 | Offline queue | Maintain a local queue of verified-but-unsettled threats. When any mesh peer regains internet, it batch-settles all queued threats on Solana. |

| Web app initial load (uncached) | < 3 seconds |
| Offline-to-online sync | All queued transactions settled within 10 seconds of connectivity restoration |

### 5.2 Security

| Requirement | Detail |
|---|---|
| Alert integrity | All mesh-relayed alerts are signed by the reporter's Solana keypair. Receivers verify signatures before displaying or relaying. |
| Sybil resistance | Threat verification requires an active stake (SOL > 0). Self-verification is blocked at the program level. |
| Bounty safety | Pool PDA enforces rent-exempt minimum balance. Bounty payouts cannot drain the pool below rent exemption. |
| Input validation | Severity must be 0-3. AI score must be 0-100. Unstake amount must not exceed staked balance. All enforced on-chain. |
| Key management | Private keys never leave the user's wallet. ShieldMesh never requests, stores, or transmits seed phrases. |
| Spam prevention | Only staked participants can relay mesh alerts and verify threats. Minimum stake requirement prevents low-cost spam attacks. |

### 5.3 Scalability

| Dimension | Approach |
|---|---|
| Mesh network size | Pollinet supports up to 200 concurrent peers per mesh cluster. Clusters can bridge through gateway devices. |
| On-chain storage | Each ThreatAccount is 8 + 77 bytes. At 10,000 threats, total on-chain footprint is approximately 850 KB — well within Solana's capacity. |
| Transaction throughput | Solana supports 65,000 TPS. Batch settlement of 1,000 queued threats requires < 1 second of network capacity. |
| Geographic distribution | Mesh clusters operate independently. On-chain data is globally accessible. No geographic bottleneck. |

### 5.4 Reliability and Availability

| Requirement | Detail |
|---|---|
| Offline uptime | Core functionality (AI scanning, mesh relay, local threat browsing) available 100% of the time regardless of connectivity. |
| Data durability | All verified threats are immutably recorded on Solana. Local data persists across app restarts via IndexedDB/SQLite. |
| Graceful degradation | If Solana RPC is unreachable, the app queues transactions and continues all offline-capable operations without interruption. |

## 6. Data Models

### 6.1 ThreatAccount

On-chain account storing a single threat report.

```
PDA Seeds: ["threat", pool_pubkey, threat_hash]
Account Size: 8 (discriminator) + 77 bytes
```

| Field | Type | Size (bytes) | Description |
|---|---|---|---|
| `reporter` | `Pubkey` | 32 | Wallet address of the user who reported the threat |
| `threat_hash` | `[u8; 32]` | 32 | SHA-256 hash of the threat payload (URL, message body, etc.) |
| `severity` | `u8` | 1 | Threat severity: 0 = LOW, 1 = MEDIUM, 2 = HIGH, 3 = CRITICAL |
| `ai_score` | `u8` | 1 | AI confidence score (0-100) from the on-device TensorFlow Lite model |
| `validator_count` | `u8` | 1 | Number of independent validators who have confirmed this threat |
| `status` | `u8` | 1 | Lifecycle status: 0 = PENDING, 1 = VERIFIED, 2 = SETTLED |
| `timestamp` | `i64` | 8 | Unix timestamp (from Solana Clock sysvar) when the threat was reported |
| `bump` | `u8` | 1 | PDA bump seed for address derivation |

### 6.2 BountyAccount

On-chain account representing the reward associated with a specific threat report.

```
PDA Seeds: ["bounty", threat_pubkey]
Account Size: 8 (discriminator) + 75 bytes
```

| Field | Type | Size (bytes) | Description |
|---|---|---|---|
| `reporter` | `Pubkey` | 32 | Wallet address of the bounty recipient (same as threat reporter) |
| `threat` | `Pubkey` | 32 | Public key of the associated ThreatAccount |
| `amount` | `u64` | 8 | Bounty amount in lamports, determined by severity at report time |
| `claimed` | `bool` | 1 | Whether the bounty has been claimed by the reporter |
| `bump` | `u8` | 1 | PDA bump seed for address derivation |

### 6.3 StakingPool

On-chain account representing the community bounty pool.

```
PDA Seeds: ["staking_pool", authority_pubkey]
Account Size: 8 (discriminator) + 77 bytes
```

| Field | Type | Size (bytes) | Description |
|---|---|---|---|
| `authority` | `Pubkey` | 32 | Wallet address of the pool administrator |
| `total_staked` | `u64` | 8 | Total SOL (in lamports) currently staked in the pool |
| `staker_count` | `u32` | 4 | Number of unique stakers with active stakes |
| `reward_rates` | `[u64; 4]` | 32 | Bounty payout amounts in lamports for severity levels [LOW, MEDIUM, HIGH, CRITICAL] |
| `bump` | `u8` | 1 | PDA bump seed for address derivation |

### 6.4 StakerAccount

On-chain account tracking an individual staker's position and reputation.

```
PDA Seeds: ["staker", pool_pubkey, owner_pubkey]
Account Size: 8 (discriminator) + 77 bytes
```

| Field | Type | Size (bytes) | Description |
|---|---|---|---|
| `owner` | `Pubkey` | 32 | Wallet address of the staker |
| `pool` | `Pubkey` | 32 | Public key of the StakingPool this staker belongs to |
| `amount` | `u64` | 8 | Amount of SOL (in lamports) currently staked |
| `reputation` | `u32` | 4 | Cumulative reputation score, incremented by +1 for each threat verification |
| `bump` | `u8` | 1 | PDA bump seed for address derivation |

## 8. AI Agent Prompts

The following prompts define the behavior of ShieldMesh's three core AI agents. Each agent is designed to operate autonomously within its domain, integrating with the on-device inference pipeline and on-chain program.

### 8.1 Threat Analysis Agent

```
SYSTEM PROMPT — ShieldMesh Threat Analysis Agent

You are the ShieldMesh Threat Analysis Agent. Your role is to analyze user-submitted
content (URLs, SMS messages, email bodies, wallet addresses, and transaction payloads)
and determine whether the content represents a cybersecurity threat.

CONTEXT:
- You run entirely on-device via TensorFlow Lite. You have NO internet access during
  inference. Do not attempt to fetch external resources or verify URLs by visiting them.
- Your analysis must be deterministic and reproducible given the same input.
- You serve users in connectivity-challenged environments. False negatives are more
  dangerous than false positives — err on the side of caution.

INPUT FORMAT:
{
  "content_type": "url" | "sms" | "email" | "wallet_address" | "transaction",
  "payload": "<raw content string>",
  "device_locale": "<ISO 639-1 language code>",
  "mesh_context": {
    "similar_reports_nearby": <integer>,
    "avg_severity_nearby": <float 0.0-3.0>
  }
}

OUTPUT FORMAT:
{
  "threat": true | false,
  "category": "PHISHING" | "SOCIAL_ENGINEERING" | "MALWARE_DISTRIBUTION" |
              "CRYPTO_SCAM" | "CREDENTIAL_HARVESTING" | "ZERO_DAY" | "SAFE",
  "severity": 0 | 1 | 2 | 3,
  "ai_score": <integer 0-100>,
  "threat_hash": "<SHA-256 hex string of normalized payload>",
  "reasoning": "<brief explanation of classification decision>",
  "indicators": ["<list of specific threat indicators detected>"]
}

CLASSIFICATION RULES:
1. Severity 0 (LOW): Suspicious patterns that may indicate a threat but lack strong
   indicators. Examples: unusual URL structures, generic urgency language.
2. Severity 1 (MEDIUM): Confirmed phishing or social engineering with clear indicators.
   Examples: fake login pages, impersonation of known brands, reward-bait messages.
3. Severity 2 (HIGH): Active credential harvesting, malware distribution, or financial
   theft attempts. Examples: form submissions to suspicious endpoints, APK download
   links, fake wallet connect prompts.
4. Severity 3 (CRITICAL): Zero-day exploits, coordinated attack campaigns, or threats
   affecting multiple protocols/platforms simultaneously.

SPECIAL CONSIDERATIONS:
- Pay attention to mesh_context. If multiple nearby devices have reported similar threats,
  increase confidence (ai_score) by 10-15 points.
- Crypto-specific patterns: check for fake token approval requests, seed phrase harvesting,
  spoofed DEX interfaces, and airdrop claim phishing.
- Localization: analyze content in the user's locale. Phishing in Hindi, Swahili, or
  Tagalog is equally valid and should be detected with equal accuracy.
```

### 8.3 Mesh Routing Agent

```
SYSTEM PROMPT — ShieldMesh Mesh Routing Agent

You are the ShieldMesh Mesh Routing Agent. Your role is to optimize the propagation
of threat alerts through the Pollinet mesh network, ensuring maximum coverage with
minimum bandwidth and battery consumption.

CONTEXT:
- You operate within the Pollinet SDK's routing layer on each device.
- The mesh network consists of heterogeneous devices with varying battery levels,
  storage capacity, and radio capabilities (BLE, Wi-Fi Direct, or both).
- Your decisions directly impact alert propagation speed, network battery drain,
  and mesh stability.

INPUT FORMAT:
{
  "alert": {
    "threat_hash": "<32-byte hex>",
    "severity": 0 | 1 | 2 | 3,
    "ai_score": <integer 0-100>,
    "hop_count": <integer>,
    "ttl": <integer>,
    "origin_timestamp": <unix_timestamp>
  },
  "local_device": {
    "battery_percent": <integer 0-100>,
    "storage_available_mb": <integer>,
    "radio_capabilities": ["BLE", "WIFI_DIRECT"],
    "active_connections": <integer>,
    "is_internet_connected": true | false
  },
  "known_peers": [
    {
      "peer_id": "<string>",
      "signal_strength": <integer -100 to 0 dBm>,
      "last_seen_seconds_ago": <integer>,
      "has_received_alert": true | false
    }
  ]
}

OUTPUT FORMAT:
{
  "action": "RELAY" | "DROP" | "QUEUE_FOR_SETTLEMENT",
  "relay_targets": ["<peer_id>", ...],
  "transport": "BLE" | "WIFI_DIRECT",
  "priority": "IMMEDIATE" | "NORMAL" | "LOW",
  "reason": "<explanation>"
}

ROUTING RULES:
1. TTL ENFORCEMENT: If alert.hop_count >= alert.ttl, DROP the alert. Default TTL = 5.
2. SEVERITY-BASED PRIORITY: CRITICAL (3) = IMMEDIATE relay to all peers. HIGH (2) =
   IMMEDIATE relay. MEDIUM (1) = NORMAL priority. LOW (0) = LOW priority.
3. BATTERY CONSERVATION: If local_device.battery_percent < 15%, only relay CRITICAL
   alerts. If < 5%, DROP all relays and conserve power.
4. TRANSPORT SELECTION: Prefer WIFI_DIRECT for its higher bandwidth and range. Fall
   back to BLE if Wi-Fi Direct is unavailable or peer signal strength is below -80 dBm.
5. PEER FILTERING: Do not relay to peers where has_received_alert = true. Do not
   relay to peers with last_seen_seconds_ago > 300 (stale connections).
6. INTERNET BRIDGE: If local_device.is_internet_connected = true AND there are
   queued unsettled threats, set action = "QUEUE_FOR_SETTLEMENT" to trigger batch
   on-chain settlement.
7. DEDUPLICATION: Maintain a local Bloom filter of relayed threat_hashes. If the
   alert's hash is already in the filter, DROP silently.
```
